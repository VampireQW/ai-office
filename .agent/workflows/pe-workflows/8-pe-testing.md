# 🧪 测试用例工作流

## 🧠 AIky 集成

> 开始前，先阅读以下 AIky 知识：
> - `AIky/identity.md` — 统一身份档案；重点读取"质量优先"、"细节控"、"有据可依"、"场景回归法"
>
> **测试风格要求**：
> - 测试用例必须可执行、可验证，不接受模糊描述
> - 覆盖正常流程 + 异常流程 + 边界场景
> - 每个核心功能至少 3 个测试用例（正常/异常/边界各一）

## 适用场景
- 代码实现完成后的功能验收
- 交付前的质量检查
- 迭代优化后的回归验证

## 执行步骤

### 1. 梳理测试范围
基于 PRD（`3-PRD.md`）、架构设计（`4-架构设计.md`）、开发任务（`5-开发任务.md`）和代码实现（`6-代码实现/`），梳理：
- 需要测试的功能模块
- 需要验证的交互流程
- 需要检查的边界条件
- 需要覆盖的 `AC-*` 和 `NFR-*`

### 2. 编写测试用例
按以下模板为每个功能模块编写用例：

#### 用例模板

| 字段 | 说明 |
|------|------|
| **用例ID** | TC-[模块]-[序号]，如 TC-LOGIN-001 |
| **模块** | 所属功能模块 |
| **规格来源** | FR-001 / NFR-001 |
| **验收标准** | AC-001 |
| **关联任务** | TASK-001 |
| **用例标题** | 一句话描述测试内容 |
| **前置条件** | 执行前需要满足的条件 |
| **操作步骤** | 详细的操作步骤（1、2、3...） |
| **预期结果** | 具体的预期表现 |
| **优先级** | P0（必测）/ P1（重要）/ P2（一般） |
| **测试类型** | 正常/异常/边界 |

### 3. 功能测试用例
为每个核心功能编写：

#### 3.1 正常流程测试
- 核心 Happy Path 完整走通
- 每个页面/视图正常加载
- 数据展示正确（列表、详情、统计）
- CRUD 操作完整闭环

#### 3.2 异常流程测试
- 必填字段未填提交
- 无效数据输入（格式错误、超范围）
- 网络异常模拟（如有）
- 权限不足操作（如有角色区分）

#### 3.3 边界场景测试
- 空数据场景（首次使用/无数据）
- 大量数据场景（列表超长/翻页）
- 极端值输入（超长文本、特殊字符、最大最小值）
- 快速重复操作（连续点击提交）

### 4. 交互验证清单
逐一验证每个可交互元素：

| 元素类型 | 验证项 | 检查内容 |
|---------|--------|---------|
| 按钮 | hover 状态 | 有视觉反馈（颜色/阴影变化） |
| 按钮 | click 反馈 | 有 active 态 / loading 态 |
| 按钮 | disabled 态 | 条件不满足时不可点击 |
| 链接/标签 | 点击跳转 | 正确切换页面/视图 |
| 表单 | focus 样式 | 输入框聚焦时有高亮 |
| 表单 | 校验提示 | 错误时有红色提示文字 |
| Modal | 打开/关闭 | 有动画过渡，点遮罩可关闭 |
| Toast | 自动消失 | 3s 后自动消失 |
| 列表 | 空态 | 无数据时展示空态页 |
| 搜索 | 无结果 | 搜索无匹配时有提示 |

### 5. 浏览器兼容性检查
// turbo
使用 browser_subagent 分别在不同窗口大小下测试：
- **Desktop**（1280px）: 布局、功能完整
- **Tablet**（768px）: 布局自适应
- **Mobile**（375px）: 核心功能可用

### 6. 性能基础检查
- 页面首次加载耗时合理（<3s）
- 无 JavaScript 控制台报错
- 无明显卡顿（大列表滚动流畅）
- 图片/字体资源加载正常

### 7. 执行测试并记录结果
使用 browser_subagent 逐一执行测试用例：
- 截图关键测试步骤
- 记录 Pass / Fail 状态
- Fail 的用例记录实际结果和复现步骤
- 将测试结果回写到 `5-开发任务.md` 的追踪矩阵
- P0 失败必须生成 `FIX-*` 并回到 `/pe-code` 修复
- AC 不可验证必须生成 `CHANGE-*` 并回到 `/pe-change`

### 8. SDD 覆盖率门槛

| 对象 | 门槛 | 不满足时处理 |
|------|------|--------------|
| P0 AC | 100% 有 TC 覆盖，100% Pass | 必须生成 `FIX-*` 或 `CHANGE-*` |
| P1 AC | 100% 有 TC 覆盖 | Fail/Blocked 必须说明原因和后续动作 |
| P2 AC | 尽量覆盖 | 未覆盖需说明是否延期 |
| NFR | 至少有验证记录 | 生成质量测试或风险记录 |
| DATA | 正常/空态/异常/边界均需验证 | 回到 `/pe-tasks` 或 `/pe-code` 补齐 |

  ### 2.1 [模块A] 测试用例
  ### 2.2 [模块B] 测试用例

## 四、SDD 覆盖率
| 需求ID | 验收标准ID | 测试用例ID | 结果 | 备注 |
|--------|------------|------------|------|------|

## 五、兼容性测试结果

## 六、缺陷清单（如有）
| 缺陷ID | 严重度 | 描述 | 复现步骤 | 状态 |

## 测试原则
1. **可执行**: 每个用例有明确的操作步骤
2. **可验证**: 每个用例有具体的预期结果
3. **高覆盖**: 正常 + 异常 + 边界全覆盖
4. **可追溯**: 用例 ID 对应到 PRD 的 FR/AC/NFR 编号和 TASK 编号
5. **覆盖率**: P0/P1 的 AC 必须 100% 有测试用例覆盖
6. **失败闭环**: P0 Fail 必须生成 FIX，AC 不可验证必须生成 CHANGE
