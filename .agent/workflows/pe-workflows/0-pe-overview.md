# 🚀 产研工作流体系（Product & Engineering）

## 工作流概览

本工作流覆盖独立软件项目从 0 到 1 的完整研发链路，7 大阶段闭环交付：

```
市场调研 → 竞品分析 → PRD文档 → 架构设计 → SDD任务拆解 → 代码实现 → 测试用例
```

> **定位升级**：不是"画原型"，而是**做产品**。代码阶段直接产出可交付的 1.0 版本，
> 具备丰富的演示标准数据和完整的分支交互流程。

## 各阶段工作流

| 阶段 | 命令 | 说明 | 产出物 |
|------|------|------|--------|
| 1. 市场调研 | `/pe-research` | 市场规模、用户画像、技术可行性 | `1-调研报告.md` |
| 2. 竞品分析 | `/pe-competitive` | 竞品体验、功能对比、差异化策略 | `2-竞品分析.md` |
| 3. PRD文档 | `/pe-prd` | 完整需求文档、产品框架图、验收标准 | `3-PRD.md` |
| 4. 架构设计 | `/pe-architecture` | 信息架构、技术选型、数据库/API设计 | `4-架构设计.md` |
| 5. SDD任务拆解 | `/pe-tasks` | 规格编号、任务拆解、追踪矩阵 | `5-开发任务.md` |
| 6. 代码实现 | `/pe-code` | **可交付 1.0 版本**（非DEMO） | `6-代码实现/` |
| 7. 测试用例 | `/pe-testing` | 功能/交互/边界测试用例 | `7-测试用例.md` |
| 收尾 | `/pe-summary` | 交付状态、覆盖率、风险和结论 | `PROJECT_SUMMARY.md` |
| 变更 | `/pe-change` | Frozen 后的变更请求和影响分析 | `CHANGELOG-SDD.md` |

## 如何使用

1. **完整流程**: `/pe-full 你的产品想法` — 一键按 1→7 顺序执行全流程
2. **单独调用**: `/pe-xxx` — 调用单个阶段工作流
3. **迭代回退**: 任意阶段可回退到上游修正后重新执行

## 交付标准（区别于旧PM工作流）

### 🎯 代码交付标准
- **不是 DEMO，是 1.0 版本**：核心功能完整闭环，可直接演示和使用
- **丰富演示数据**：每个列表/表格至少 20+ 条真实数据，覆盖多种业务场景
- **完整分支流程**：正常/异常/边界三类场景全覆盖，含二次确认、错误提示、空态处理
- **四态完备**：loading / success / empty / error 状态均有对应 UI

### 🔗 SDD 规格标准
- **稳定编号**：PRD 中使用 REQ/FR/NFR/AC/DATA/RISK 编号
- **规格冻结**：代码实现前 PRD 必须冻结为 SPEC 基线
- **任务追踪**：每个 TASK 必须关联规格来源和验收标准
- **测试回溯**：每个 TC 必须对应 AC 或 NFR
- **变更控制**：Frozen 后的新需求必须登记为 CHANGE
- **交付总结**：交付前生成 PROJECT_SUMMARY，汇总覆盖率和风险
