# 🚀 产研完整工作流

## 🧠 AIky 集成

> 每个阶段的工作流都已内置 AIky 身份和方法论引用。
> 执行时 AI 会读取 `AIky/identity.md` 中的统一身份档案、思维模型、工作风格和工作原则，
> 确保整个流程输出符合 ky 的质量标准和思维风格。

## 适用场景
- 从零开始的独立软件项目
- 需要完整产研交付物的项目
- 个人独立项目的完整研发流程

## 执行前准备

请提供以下信息：
- **产品是什么？** 解决什么问题？
- **目标用户是谁？** 核心使用场景？
- **有什么参考产品？** 竞品或灵感来源？
- **技术偏好？** 纯前端 / 全栈 / 移动端？

## 执行流程

### 第一步：市场调研
执行 `/pe-research` 工作流
- 市场规模分析、用户画像、技术可行性预研
- 产出：`1-调研报告.md`
- ⏸️ **暂停确认**：确认调研方向和发现是否准确

### 第二步：竞品分析
执行 `/pe-competitive` 工作流
- 竞品识别与深度体验、功能对比、差异化策略
- 产出：`2-竞品分析.md`
- ⏸️ **暂停确认**：确认竞品选择和差异化方向

### 第三步：PRD文档
执行 `/pe-prd` 工作流
- 完整需求文档、产品框架图、功能规则、验收标准、规格编号
- 产出：`3-PRD.md`
- ⏸️ **暂停确认**：确认需求范围、优先级和 SPEC 基线，冻结 PRD

### 第四步：架构设计
执行 `/pe-architecture` 工作流
- 基于 Frozen PRD 设计信息架构、功能模块、技术选型、数据库/API
- 产出：`4-架构设计.md`
- ⏸️ **暂停确认**：确认架构方案与 Frozen PRD 一致

### 第五步：SDD任务拆解
执行 `/pe-tasks` 工作流
- 将 Frozen PRD 拆成 TASK，建立需求 → AC → 任务 → 代码 → 测试的追踪矩阵
- 产出：`5-开发任务.md`
- ⏸️ **暂停确认**：确认任务范围、实现顺序和追踪矩阵完整性

### 第六步：代码实现
执行 `/pe-code` 工作流
- **可交付的 1.0 版本**（非演示DEMO）
- 丰富演示数据（20+ 条/场景）、完整分支交互流程
- 产出：`6-代码实现/`
- ⏸️ **暂停确认**：在浏览器中演示并确认功能完整性

### 第七步：测试用例
执行 `/pe-testing` 工作流
- 功能测试、交互验证、边界场景测试
- 产出：`7-测试用例.md`
- ⏸️ **暂停确认**：确认测试覆盖度

### 最终收尾：项目总结
执行 `/pe-summary` 工作流
- 汇总 SPEC 状态、实现进度、测试覆盖率、变更记录和遗留风险
- 产出：`PROJECT_SUMMARY.md`
- ⏸️ **暂停确认**：确认交付状态为 Ready / Blocked / Changed

## 质量红线

> **代码实现不是 DEMO，是 1.0 版本。** 以下标准必须满足：
> 1. 核心流程完整闭环，所有按钮/链接可点击可交互
> 2. 演示数据丰富真实：列表 20+ 条、覆盖正常/边界数据
> 3. 分支交互完整：正常流程 + 异常提示 + 空态 + 确认弹窗
> 4. 视觉品质 Premium：现代设计、动画、响应式
> 5. SDD 追踪完整：P0/P1 需求必须能追踪到 TASK、实现文件和 TC
> 6. 变更受控：Frozen 后的新需求必须走 `/pe-change`
> 7. 测试门槛：P0 AC 必须 100% 覆盖且 100% Pass
