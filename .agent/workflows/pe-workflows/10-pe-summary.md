# 📌 项目总结工作流

## 核心定位

> `/pe-summary` 用于在交付前生成 `PROJECT_SUMMARY.md`，让项目状态一页可读。
> 它不是宣传页，而是产研交付状态面板。

## 输入前提

- `1-调研报告.md`
- `2-竞品分析.md`
- `3-PRD.md`
- `4-架构设计.md`
- `5-开发任务.md`
- `6-代码实现/`
- `7-测试用例.md`
- `CHANGELOG-SDD.md`（如有变更）

## 执行步骤

### 1. 汇总规格状态

从 `3-PRD.md` 提取：
- 当前 SPEC 版本
- PRD 状态：Draft / Frozen / Changed
- P0/P1/P2 需求数量
- 核心用户需求 `REQ-*`
- 核心功能需求 `FR-*`
- 非功能需求 `NFR-*`

### 2. 汇总实现状态

从 `5-开发任务.md` 提取：
- TASK 总数
- Implemented / Tested / Blocked / Changed 数量
- 未完成任务及原因
- 涉及的实现文件

### 3. 汇总测试状态

从 `7-测试用例.md` 提取：
- 测试用例总数
- Pass / Fail / Blocked 数量
- P0 AC 覆盖率和通过率
- P1 AC 覆盖率和通过率
- NFR 验证结果
- 缺陷清单摘要

### 4. 汇总变更状态

从 `CHANGELOG-SDD.md` 提取：
- Accepted / Rejected / Deferred 变更数量
- 当前 SPEC 版本演进
- 未关闭变更项

### 5. 输出 PROJECT_SUMMARY.md

在 `独立项目/[项目名]/PROJECT_SUMMARY.md` 生成：

```markdown
# [项目名] 项目总结

| 信息项 | 内容 |

### 2.2 核心问题
### 2.3 北极星指标
```

## 质量门槛

- [ ] 结论必须基于 PRD、任务、测试和变更记录，不主观臆断
- [ ] 所有数字必须能追溯到源文档
- [ ] Blocked / Failed / Changed 项必须列出原因
- [ ] 交付状态只能是 Ready / Blocked / Changed 之一
