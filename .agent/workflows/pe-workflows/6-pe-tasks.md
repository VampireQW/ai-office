# 📋 SDD 任务拆解工作流

## 核心定位

> `/pe-tasks` 是 PRD/架构和代码之间的 SDD 桥梁。
> 它不新增需求，只把已冻结的规格和已确认的架构拆成可执行任务，确保每个开发动作都有明确的规格来源。

## 输入前提

- `3-PRD.md` 已完成
- `4-架构设计.md` 已完成
- PRD 状态为 `Frozen`
- P0/P1 功能需求已有稳定编号：`FR-001`
- 验收标准已有稳定编号：`AC-001`
- 数据规格、非功能需求已有编号：`DATA-001`、`NFR-001`
- PRD 中的 SDD 追踪矩阵已初始化

## 执行步骤

### 1. 读取规格基线

读取 `3-PRD.md`，提取：
- 功能需求：`FR-*`
- 用户需求：`REQ-*`
- 验收标准：`AC-*`
- 非功能需求：`NFR-*`
- 数据规格：`DATA-*`
- 风险约束：`RISK-*`
- 当前规格版本：`SPEC-v*`

读取 `4-架构设计.md`，提取：
- 信息架构
- 功能模块与规格来源映射
- 数据模型
- API / 接口设计
- 非功能性设计

如发现 P0/P1 需求缺少编号或验收标准，先返回 `/pe-prd` 补齐，不进入开发任务拆解。
如发现架构模块无法追溯到 `FR/AC/NFR/DATA`，先返回 `/pe-architecture` 修正。

### 2. 拆解实现任务

按以下粒度拆解任务：
- 一个任务应能在单次开发中完成
- 一个任务必须关联至少一个 `FR` 或 `NFR`
- 一个任务必须关联至少一个 `AC`
- 数据任务必须关联 `DATA`
- 不允许出现没有规格来源的任务

任务编号格式：

| 类型 | 编号格式 | 说明 |
|------|----------|------|
| 开发任务 | TASK-001 | 页面、组件、状态、逻辑、数据实现 |
| 测试任务 | TEST-001 | 自动化测试、人工验收、兼容性验证 |
| 修复任务 | FIX-001 | 测试失败或缺陷修复 |

### 3. 定义任务模板

每个任务必须使用以下结构：

```markdown
### TASK-001 [任务名称]
**规格来源**: FR-001 / NFR-001 / DATA-001
**验收标准**: AC-001, AC-002
**优先级**: P0/P1/P2
**任务类型**: UI / Logic / Data / State / Integration / Quality
**预计修改文件**:
- `6-代码实现/index.html`
- `6-代码实现/css/style.css`
- `6-代码实现/js/app.js`

**实现要求**:
1. 要求1
2. 要求2

**完成定义（DoD）**:
- [ ] 对应 AC 已实现
- [ ] 四态/异常/边界已覆盖
- [ ] 无控制台错误
- [ ] 已更新追踪矩阵
```

### 4. 生成开发顺序

按依赖关系排序：
1. 基础数据与状态模型
2. 页面结构与导航
3. 核心 P0 流程
4. 表单、CRUD、搜索筛选
5. 四态与异常分支
6. 响应式、视觉品质、微交互
7. 非功能需求与质量检查

### 5. 更新 SDD 追踪矩阵

在任务文档中维护完整追踪表：

| 需求ID | 验收标准ID | 任务ID | 实现文件 | 测试用例ID | 状态 |
|--------|------------|--------|----------|------------|------|
| FR-001 | AC-001 | TASK-001 | `js/app.js` | TC-[模块]-001 | Planned |

状态枚举：
- `Planned`
- `In Progress`
- `Implemented`
- `Tested`
- `Changed`
- `Blocked`
- `Deprecated`

### 6. 输出任务文档

在 `独立项目/[项目名]/5-开发任务.md` 生成文档，包含：

```markdown
# [项目名] SDD 开发任务

## 一、规格基线
| 信息项 | 内容 |
|--------|------|
| PRD 文件 | `3-PRD.md` |
| 架构文件 | `4-架构设计.md` |
| 规格版本 | SPEC-v1.0 |
| PRD 状态 | Frozen |
| 生成日期 | [日期] |

## 二、任务总览
| 任务ID | 名称 | 规格来源 | 验收标准 | 优先级 | 状态 |
|--------|------|----------|----------|--------|------|

## 三、开发任务详情

## 四、测试任务建议

## 五、SDD 追踪矩阵

## 六、变更控制
| 日期 | 变更项 | 影响需求 | 影响任务 | 影响测试 | 状态 |
|------|--------|----------|----------|----------|------|
```

## 质量门槛

- [ ] 所有 P0/P1 `FR` 至少对应一个 `TASK`
- [ ] 所有 P0/P1 `REQ` 至少对应一个 `FR`
- [ ] 所有 `AC` 至少对应一个 `TASK` 或 `TEST`
- [ ] 所有 `DATA` 至少对应一个数据实现任务
- [ ] 所有 `NFR` 至少对应一个质量任务或测试任务
- [ ] 不存在无规格来源的开发任务
- [ ] 所有 `FIX-*` 都关联失败的 `TC-*` 或缺陷记录
- [ ] 任务状态和追踪矩阵一致

## 下一步

任务拆解完成后，使用 `/pe-code` 进入代码实现阶段。
