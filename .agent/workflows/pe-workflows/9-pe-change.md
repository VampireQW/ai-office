# 🔁 SDD 变更控制工作流

## 核心定位

> `/pe-change` 用于处理 Frozen PRD 之后出现的任何新需求、范围调整、验收标准变化或实现偏差。
> 变更不能直接进入代码，必须先完成影响分析和规格同步。

## 触发场景

- 用户在实现中提出新功能或改动
- 开发阶段发现 PRD 缺少必要规则
- 测试阶段发现 AC 不可验证或覆盖不足
- 技术实现要求调整架构、数据模型或交互流程
- 需要延期、废弃或降级某个 `FR/AC/NFR/DATA/TASK/TC`

## 执行步骤

### 1. 登记变更请求

生成稳定变更编号：

| 类型 | 编号格式 | 说明 |
|------|----------|------|
| 变更请求 | CHANGE-001 | 新增、修改、删除、延期、废弃 |
| 缺陷修复 | FIX-001 | 测试失败或实现偏差导致的修复 |

变更请求模板：

```markdown
### CHANGE-001 [变更标题]
**提出来源**: 用户 / 开发 / 测试 / 技术约束
**变更类型**: Add / Modify / Remove / Defer / Deprecate
**变更描述**: [具体说明]
**触发原因**: [为什么现在需要变更]
**影响对象**: REQ-001 / FR-001 / AC-001 / NFR-001 / DATA-001 / TASK-001 / TC-001
**优先级**: P0/P1/P2
**决策**: Accepted / Rejected / Deferred
**决策理由**: [明确理由]
```

### 2. 做影响分析

逐项检查变更影响：

| 影响域 | 检查问题 |
|--------|----------|
| 用户需求 | 是否新增或改变 `REQ-*` |
| 功能需求 | 是否新增、修改、废弃 `FR-*` |
| 验收标准 | 是否需要新增或修改 `AC-*` |
| 非功能需求 | 是否影响 `NFR-*` |
| 数据规格 | 是否影响 `DATA-*` |
| 架构设计 | 是否影响模块、数据模型、API、权限 |
| 开发任务 | 是否需要新增 `TASK-*` 或 `FIX-*` |
| 测试用例 | 是否需要新增、修改或废弃 `TC-*` |

### 3. 更新规格基线

如果变更被接受：
- 更新 `3-PRD.md` 的相关 `REQ/FR/AC/NFR/DATA/RISK`
- 如影响技术方案，同步更新 `4-架构设计.md`
- 在 PRD 变更记录中写入 `CHANGE-*`
- 将 PRD 状态从 `Frozen` 改为 `Changed`
- 形成新规格基线，例如 `SPEC-v1.1`
- 用户确认后重新标记为 `Frozen`

如果变更被拒绝：
- 记录拒绝理由
- 不修改 PRD、任务和代码

如果变更延期：
- 标记为 `Deferred`
- 放入后续版本规划
- 不进入当前实现范围

### 4. 同步任务和测试

更新 `5-开发任务.md`：
- 新增或修改 `TASK-*`
- 测试失败引起的修复新增 `FIX-*`
- 更新 SDD 追踪矩阵状态为 `Changed`
- 废弃项标记为 `Deprecated`，不删除历史编号

更新 `7-测试用例.md`：
- 新增或修改对应 `TC-*`
- 若 AC 改变，相关 TC 必须重新执行
- 失败修复完成后重新标记测试结果

### 5. 输出变更记录

在 `独立项目/[项目名]/CHANGELOG-SDD.md` 追加记录：

```markdown
# [项目名] SDD 变更记录

## CHANGE-001 [变更标题]
| 信息项 | 内容 |
|--------|------|
| 日期 | [日期] |
| 来源 | 用户 / 开发 / 测试 |
| 类型 | Add / Modify / Remove / Defer / Deprecate |
| 决策 | Accepted / Rejected / Deferred |
| 规格版本 | SPEC-v1.0 → SPEC-v1.1 |

## 影响分析
| 对象类型 | 影响项 | 处理方式 |
|----------|--------|----------|
| PRD | FR-001 / AC-001 | 修改 |
| 任务 | TASK-001 / FIX-001 | 新增/修改 |
| 测试 | TC-001 | 重新执行 |

## 决策理由

## 后续动作
```

## 质量门槛

- [ ] 每个 Accepted 变更都更新了 PRD
- [ ] 每个 PRD 变更都更新了任务追踪矩阵
- [ ] 每个 AC/NFR 变更都更新了测试用例
- [ ] 不存在直接进入代码但没有 `CHANGE-*` 的新需求
- [ ] 废弃编号标记为 Deprecated，没有被复用
- [ ] SPEC 版本变化清晰可追溯

## 下一步

- Accepted：回到 `/pe-tasks` 同步任务，再进入 `/pe-code`
- Rejected：继续当前流程
- Deferred：进入后续版本规划
